## 2026-06-25 — 전역 Python 환경 오염 (numpy 버전 충돌)

**문제:** 평가 하니스를 돌리려고 `naengchae-langchain/requirements.txt`(langgraph, faiss-cpu 등)를
별 생각 없이 시스템 전역 conda 환경에 `pip install`했다. 그 결과 `numpy`가 2.2.6으로 올라가면서,
같은 환경에 이미 설치돼 있던 matplotlib/pandas/scikit-learn/scipy(전부 `numpy<2` 요구)가 전부
깨질 수 있는 상태가 됐다.

**어떻게 알았나:** `pip install` 직후 출력에 `ERROR: pip's dependency resolver does not currently
take into account all the packages...` 경고가 떴고, 이걸 무시하지 않고 바로
`python3 -c "import numpy; print(numpy.__version__)"`로 실제 버전을 확인해 문제를 확정했다.

**전략 고민:** 두 가지 선택이 있었다.
1. numpy만 다시 `<2`로 내려서 봉합한다 → 당장은 해결되지만, 이 프로젝트가 또 자기 패키지를
   깔 때마다 전역 환경을 건드리는 구조적 문제가 남는다.
2. 이 프로젝트를 위한 **격리된 가상환경(`.venv`)**을 새로 만들고, 전역 환경은 원래 상태로
   복구한다 → 근본적으로 재발을 막는다.

**해결:** 2번을 선택. `naengchae-langchain/.venv`를 만들어 의존성을 전부 그 안으로 옮기고,
전역 conda 환경은 `numpy==1.26.4`로 복구 + 잘못 깔린 langgraph 등 패키지를 전부 제거해
원상복구했다. 이후 모든 평가 실행은 `.venv` 안에서만 수행.

**배운 점:** 이 머신은 여러 프로젝트(Resume wiki 등)가 같은 전역 Python 환경을 공유하고 있었다.
"새 프로젝트 = 새 가상환경"을 처음부터 원칙으로 삼았어야 했고, 이후 모든 Python 작업에 이
원칙을 적용하기로 했다.

## 2026-06-25 — 검증기와 RAG 지식베이스가 서로 모순되는 버그

**문제:** 평가 하니스로 24개 합성 케이스를 돌렸을 때 `microwave`(전자레인지 전용) 환경 케이스
4건 중 3건이 실패했다. 실패 사유는 한결같이 "찜 조리법을 추천할 수 없습니다"였다.

**어떻게 알았나:** 단순히 "실패율이 높다"에서 멈추지 않고, `eval/run_eval.py`가 실패 사유
문자열을 카테고리별로 자동 집계하도록 설계해뒀기 때문에 — `forbidden_cooking_method` 카테고리에
microwave 케이스들이 집중되는 패턴이 바로 보였다. 패턴을 발견한 뒤 `graph.py`의
`ENV_FORBIDDEN_KEYWORDS["microwave"]`를 열어보니 `"찜"`이 금지 목록에 들어 있었는데,
같은 프로젝트의 `knowledge_base.py`(RAG 지식베이스)는 "전자레인지 환경에서는 계란찜·단호박찜이
적합하다"고 정반대로 명시하고 있었다.

**전략 고민:** 둘 중 하나가 틀렸다는 뜻인데, 어느 쪽을 신뢰할지 판단해야 했다. 실제 전자레인지로
계란찜을 만들 수 있는지(상식/사실관계)를 기준으로, 지식베이스 쪽이 맞고 검증기 쪽이 틀렸다고
결론. LLM이 지식베이스를 참고해서 올바르게 추천해도 결정론적 검증기가 무조건 막아버리는
구조였다.

**해결:** `ENV_FORBIDDEN_KEYWORDS["microwave"]`에서 `"찜"`만 제거. (다른 금지 키워드인
오븐/에어프라이어/가스레인지/프라이팬/직화/팬은 그대로 유지 — 실제로 전자레인지로 불가능한
조리법이므로.)

**배운 점:** "검증 로직이 있다"는 것과 "검증 로직이 맞다"는 것은 다르다. 결정론적 검증기도
버그가 날 수 있고, 그걸 잡으려면 검증기 자체가 참조하는 다른 자료(RAG 지식베이스)와 대조해보는
습관이 필요하다. 평가 하니스가 실패율만 보여줬다면 "프롬프트가 부족하다"고 잘못 결론 내리고
프롬프트만 계속 고쳤을 수도 있었는데, 실패를 유형별로 집계해둔 덕분에 진짜 원인(검증기 버그)을
찾을 수 있었다.

## 2026-06-25 — "베이스라인 → 개선 → 재측정"을 몇 번 반복하고 언제 멈출지 판단하기

**문제:** 프롬프트를 고치면 통과율이 오르긴 하는데, 끝없이 고칠 수도 있는 작업이다. 언제
"충분하다"고 판단해야 하는지 기준이 없으면 한쪽은 과소(버그를 놔둔 채 끝), 다른 한쪽은
과다(수확이 거의 없는데도 계속 프롬프트를 누적시켜 프롬프트가 비대해지는) 위험이 있었다.

**전략:** 매 라운드마다 "이번에 고친 게 검증기 버그인가, 프롬프트 누락인가, 아니면 LLM이
산발적으로 같은 실수를 반복하는 것인가"를 구분해서 보기로 했다.
- 검증기 버그 → 명확한 근본 원인이라 고치면 확실히 좋아진다 (1라운드: 70.8%→79.2%)
- 프롬프트에 명시되지 않은 규칙(암묵적 재료 가정, 유통기한 플래그 재확인) → 구체적 예시를
  넣으면 크게 좋아진다 (2라운드: 79.2%→91.7%)
- 남은 2건은 같은 두 유형이 산발적으로 재발하는 경우였는데, 이건 프롬프트 규칙을 더 늘려도
  100% 박멸이 보장되지 않는 영역(LLM의 확률적 추론 오류)이라고 판단해, 매 라운드 전/후 수치를
  사용자에게 보여주고 "여기서 멈출지, 더 팔지"를 직접 확인받는 방식으로 진행했다.

**해결/결론:** 2라운드(91.7%) 후 사용자 승인을 받고 중단. 추가 개선이 필요해지면 few-shot
예시나 generate 후 self-critique 단계 같은 더 무거운 기법이 필요하다는 점을 다음 단계로
남겨뒀다 (`naengchae-langchain/eval/README.md` 참고).

**배운 점:** "측정→개선"을 반복할 때, 매번 "이건 근본 원인 수정인가 산발적 오류 완화인가"를
구분해두면 언제 멈춰야 하는지 판단 기준이 생긴다. 그리고 그 판단을 혼자 내리지 않고 매 라운드
사용자에게 수치를 보여주고 확인받았다.

## 2026-06-25 — Git 히스토리 충돌 위기: 로컬엔 `.git`이 없는데 실제 GitHub엔 6개 커밋이 있었다

**문제:** Phase 1 작업물을 GitHub에 올리려고 보니, 로컬 작업 폴더에는 `.git` 폴더가 전혀 없었다.
"새 프로젝트니까 `git init`하면 되겠다"고 생각하고 새로 초기화한 뒤 모든 파일을 하나의 root
커밋으로 만들었다. 그런데 사용자가 보여준 GitHub 스크린샷을 보니 실제 저장소
(`Eclipse-Universe/Naengchae`)에는 이미 6개 커밋(2주 전부터 시작, `assets/` 폴더 포함)의 진짜
히스토리가 있었다. 그대로 push했다면 push가 거부되거나, force-push를 했다면 기존 6개 커밋의
히스토리를 통째로 날릴 뻔한 상황이었다.

**전략:** 절대 바로 push하지 않고, 먼저 **읽기 전용으로만** 실제 상태를 확인하기로 했다.
1. `git remote add` + `git fetch`만 실행 (push 안 함)
2. `git log origin/main`으로 실제 히스토리, `git ls-tree -r origin/main`으로 실제 파일 목록을
   확인 → 로컬에 없는 `assets/` 폴더(아이콘 6개)가 원격에는 있다는 걸 발견
3. 공통 파일들(`package.json`, `app.json` 등)을 `git diff origin/main -- <file>`로 하나씩
   비교해서, 정말 내가 새로 만든 파일(`naengchae-langchain/`, `README.md`)과 의도적으로 고친
   파일(`.gitignore`)만 다르고 나머지는 동일함을 확인
4. 로컬에서 만든 단독 커밋을 백업 브랜치(`local-snapshot-before-history-merge`)로 보존한 뒤,
   `git checkout -b main origin/main`으로 진짜 히스토리 위로 옮겨감

**예상치 못한 부작용과 대응:** 3번 단계에서 브랜치를 전환하자 `README.md`와
`naengchae-langchain/` 전체(eval, graph.py 등)가 디스크에서 사라졌다 — 이 파일들이 백업
브랜치에만 커밋돼 있고 origin/main에는 없었기 때문에, git이 정상적으로 작업 트리에서 제거한
것이었다(데이터 손실이 아니라 git의 정상 동작). 백업 브랜치가 있었기 때문에
`git checkout local-snapshot-before-history-merge -- <경로>`로 즉시 복구했고, 복구된
`graph.py`에 Phase 1에서 고친 버그 수정 내용이 그대로 살아있는지 `grep`으로 재확인한 뒤에야
다음 단계로 진행했다.

**해결:** 진짜 히스토리(`origin/main`) 위에 Phase 1 작업물만 새 커밋으로 올리고,
`git rev-list --left-right --count origin/main...main`으로 "0 behind, 1 ahead"임을 확인한 뒤
**일반 fast-forward push**(force 없이)로 마무리.

**배운 점:** "`.git` 폴더가 없다"는 "히스토리가 없다"는 뜻이 아니다. 원격에 진짜 저장소가 있을
가능성을 먼저 확인(`fetch`)하고, 비교(`diff`)하고, 안전망(백업 브랜치)을 만든 다음에야 손을
대는 순서를 지킨 덕분에 실제 데미지 없이 끝났다. 이 일로 생긴 일반 원칙은 메모리에도
별도로 저장해뒀다 — 앞으로 어떤 프로젝트든 `.git`이 없는 디렉터리를 다룰 때 같은 순서를
따른다.

## 2026-06-25 — 과거 커밋 메시지의 옛 프로젝트명("냉파") 정리: force-push를 신중하게 쓰기

**문제:** 프로젝트는 원래 "냉파(NaengPa)"라는 이름으로 시작했다가 개발 중 "냉채(NaengChae)"로
바뀌었는데, 코드/설정에는 흔적이 전혀 없었지만 과거 커밋 메시지 2개에 "냉파"가 남아 있었다
(GitHub 히스토리 화면에서 노출됨). 포트폴리오로 보여줄 저장소이니 이것도 지우고 싶다는 요청을
받았다.

**전략 고민:** 커밋 메시지를 고치는 방법은 두 가지뿐이다 — `git rebase -i`(대화형, 이
환경에서는 도구 정책상 사용 금지) 또는 `git filter-branch --msg-filter`(비대화형, 스크립트로
메시지를 치환). 후자를 선택했는데, 이건 **그 커밋들의 SHA를 전부 바꾸고 `main`에
force-push가 필요한 작업**이라 진행 전에 사용자에게 명시적으로 확인을 받았다(영향 범위를
숫자로 설명: "7개 커밋 전체 SHA가 바뀝니다").

**예상치 못한 함정:** 처음에 짠 필터는 커밋 제목(subject)만 정확히 일치할 때 치환하도록
짰는데, 실행해보니 아무것도 안 바뀌었다. 원인을 보니 `--msg-filter`는 커밋 메시지 **전체(제목
+ 본문)**를 통째로 넘겨주는데, 실제 커밋 메시지에는 본문(불릿 목록)까지 붙어 있어서 "제목만
정확히 일치"하는 조건이 단 한 번도 맞지 않았던 것. 부분 치환(substring replace) 방식으로
다시 짜서 해결했고, "Rename app: 냉파 -> 냉채"처럼 단순 치환하면 "냉채 -> 냉채"라는 말이 안
되는 문장이 나오는 커밋은 별도로 "Rename app to 냉채(NaengChae)"로 자연스럽게 다시 썼다.

**해결:** 치환 후 `git diff <원본 ref> <새 ref>`로 **파일 내용은 한 글자도 안 바뀌었음**을
재확인 → 사용자 승인을 다시 한번 받고 force-push 실행 → 코드/커밋 메시지 양쪽 모두에서
"냉파" 흔적이 0건임을 grep으로 최종 확인.

**배운 점:** 히스토리를 다시 쓰는 작업은 (1) 영향 범위를 정확한 숫자로 설명하고 명시적 동의를
받고, (2) 실행 전에 비파괴적인 방법(메시지 필터)으로 좁히고, (3) 실행 후 파일 트리가 정말
그대로인지 diff로 검증하는 3단계를 지키면 force-push도 안전하게 쓸 수 있다.

## 2026-06-25 — Phase 2: 토큰 사용량을 얻으려면 `with_structured_output`의 `include_raw`가 필요했다

**문제:** `generate_node`는 `llm.with_structured_output(RecipeRecommendation)`을 써서 LLM
응답을 곧바로 Pydantic 객체로 받는다. 편하지만, 이 방식은 내부에서 원본 `AIMessage`(토큰
사용량이 들어있는 `usage_metadata`를 가진 객체)를 버리고 파싱된 객체만 돌려준다. 토큰/비용을
추적하려면 그 원본이 필요했다.

**전략 고민:** 두 가지를 고민했다. (1) 별도로 `llm.invoke()`를 한 번 더 호출해서 토큰만
얻는다 — LLM 호출이 2배로 늘어나 비용도 2배가 되므로 탈락. (2) LangChain의
`with_structured_output(..., include_raw=True)` 옵션을 쓴다 — 호출 1번으로
`{"raw": AIMessage, "parsed": ..., "parsing_error": ...}` 딕셔너리를 받을 수 있어 추가
비용 없이 해결된다.

**검증:** 코드를 바꾸기 전에 실제 API를 호출해서 `include_raw=True`가 정말
`raw.usage_metadata`를 채워주는지부터 확인했다(추측으로 코드를 쓰지 않기 위해). 확인 후
`generate_node`를 `result["parsed"]`(레시피)와 `result["raw"]`(토큰 추출용)로 분리했고,
`result["parsing_error"]`가 있으면 예외를 다시 던져서 기존 동작(파싱 실패 시 에러 전파)을
그대로 유지했다.

**비용 단가 확보:** 정확한 비용 계산을 위해 Upstage 공식 가격 페이지를 직접 조회했다
(`solar-pro3`: input $0.15/1M, 캐시 input $0.015/1M, output $0.6/1M tokens — 2026-06-25
확인). 추측한 가격을 쓰지 않고 실제 출처를 코드 주석과 문서에 명시했다.

**범위를 의도적으로 좁힌 부분:** RAG 검색(retrieve) 단계도 쿼리를 임베딩하느라 Upstage
Embeddings API를 호출하지만, LangChain의 `Embeddings.embed_query()`는 토큰 사용량을
반환하지 않는다. 이걸 정확히 추적하려면 LangChain 래퍼를 우회해 Upstage 클라이언트를 직접
호출해야 하는데, 비용 비중이 작은 항목(임베딩 단가 $0.10/1M, 쿼리 텍스트도 짧음) 때문에
추상화 계층을 깨는 건 과한 엔지니어링이라고 판단해 `retrieve`는 지연시간만 추적하고
토큰/비용 추적은 `generate`(실제 비용이 발생하는 LLM 호출) 단계에만 정확히 구현하기로
결정했다. 자세한 내용은 `naengchae-langchain/observability/README.md`에 명시해뒀다.

**결과로 드러난 인사이트:** 평가셋 24건을 한 번 돌리는 데 LLM 호출 37회, 토큰 76,400개,
비용 약 $0.0134가 들었다. 24건 중 평균 재시도 0.62회가 LLM 호출 수를 24회→37회로 54%
늘린 것 — 즉 Phase 1에서 했던 "재시도율을 낮추는 프롬프트 개선"이 통과율 개선일 뿐 아니라
**비용 절감**이기도 했다는 게 수치로 드러났다. Phase 1과 Phase 2가 따로 노는 작업이 아니라
서로의 효과를 증명해주는 관계라는 걸 이번에 알게 됐다.

**배운 점:** 관찰성(observability)을 나중에 붙이는 게 아니라 먼저 만들어두면, 이전 단계의
개선 작업(Phase 1의 재시도율 감소)이 가져온 효과를 다른 차원(비용)에서도 정량적으로
설명할 수 있게 된다.

## 2026-06-26 — Phase 3: "코퍼스를 늘리면 좋아진다"를 어떻게 측정할지부터 정해야 했다

**문제:** 로드맵엔 "RAG 코퍼스 확장 + retrieval 품질 평가"라고만 적혀 있었다. 실제로 손을
대기 전에 적어도 4가지를 먼저 정해야 했다: (1) 코퍼스 내용을 어디서 가져올지, (2) 얼마나
늘릴지, (3) 늘린 게 실제로 도움이 됐는지 무엇으로 증명할지, (4) "이전"과 "이후"를 어떻게
공정하게 비교할지. 이번 항목은 사용자가 명시적으로 "모든 단계·방법·전략에 이유가 있어야
한다"고 강조한 뒤 시작한 Phase라, 코드를 쓰기 전에 이 4가지에 대한 답을 먼저 정리하고
시작했다.

**(1) 데이터 소스 결정:** 실제 레시피 사이트 스크래핑(저작권/ToS 위험, 정제 비용 큼)이나
공개 데이터셋 도입(라이선스 확인·포맷 변환이라는 별도 사이클이 필요) 대신, 기존 17~18개
문서와 동일한 구조로 직접 체계적으로 작성하기로 했다. 이유: 이번 Phase의 핵심 역량은
"코퍼스 내용의 원천"이 아니라 "retrieval 품질을 어떻게 측정하고 개선하는가"이므로, 평가
방법론에 시간을 더 쓰는 게 맞다고 판단했다. 실제 외부 데이터셋 도입은 구체적인 검색
실패가 발견되면 그때 정당화되는 후속 작업으로 남겨뒀다.

**(2) 규모 결정:** 로드맵의 "수백~수천 개"보다 작은 94개(약 5.5배 확장)로 잡았다. 이유:
이 앱이 실제로 다루는 범위(조리환경 3종, 음식취향 5종, `eval/cases.json`에 실제 등장하는
재료 31종)를 전수 조사해보니, 수천 개는 거의 다 중복 패딩이 될 게 뻔했다. 가짜 재료를
늘리는 대신 이 앱이 실제로 다루는 재료부터 빠짐없이 채우는 것을 우선했다.

**(3) 측정 지표 결정:** "코퍼스가 늘면 좋아진다"를 검증하려고 처음엔 recall@k 하나로
뭉쳐서 보려 했는데, 그러면 "retriever 알고리즘이 좋아진 것"과 "코퍼스에 정보가 더 많아진
것"이 섞여서 뭐가 원인인지 설명할 수 없다는 걸 깨달았다. 그래서 두 질문으로 쪼갰다:
랭킹 품질(hit@k/recall@k/MRR, k=4 vs k=8)과 지식 커버리지 갭(이 주제가 구버전에도
있었는가)을 따로 측정했다.

**(4) Before/after 비교 방법의 함정과 수정:** 처음 계획은 "구버전 코퍼스로 같은 retriever를
돌려서 recall@k를 직접 비교"하는 것이었다. 그런데 구버전 문서엔 id가 없고, 일부 문서는
표현이 통째로 바뀌었다(예: "양파와 감자" 결합 문서 → 분리 문서, "여러 단위 유통기한
채소들..." 문서는 신버전에서 빠짐). 정확한 id 매핑이 불가능한 상태에서 억지로 매핑하면
숫자는 나오지만 그 숫자가 무엇을 측정하는지 불분명해진다는 걸 알고 멈췄다. 대신 git
히스토리(`git show HEAD:...`)에서 구버전 코퍼스 텍스트를 직접 불러와 "이 주제에 대한
키워드가 구버전 텍스트에 등장하는가"로 바꿔 측정했다. 이 방법도 완벽하지 않다 — 키워드
부분일치라서 의도와 무관한 우연한 일치가 있다(예: "1인 가구"라는 단어가 일식 설명 문서에
우연히 들어 있어서, 실제로는 없던 가구규모별 팁이 "있었다"로 잘못 잡힌다). 이 한계를
숨기지 않고 `eval/RETRIEVAL_EVAL.md`에 "이 지표는 보수적으로 과대 추정하는 방향이라
실제 신규 가치는 측정된 숫자보다 크다"고 명시했다.

**측정 결과:** k=4(현재 운영값)에서 hit_rate 1.0, recall 0.900, MRR 0.95. k=8로 늘리면
recall이 0.951까지 오르지만 컨텍스트(=Phase 2에서 측정한 비용)가 최대 2배 늘어나는 비용
대비 이득이 작아 k=4를 유지하기로 결정했다. 코퍼스 확장 자체는 38개 질의 중 최소 10건을
구버전에서 전혀 답할 수 없던 주제에서 새로 답할 수 있게 만들었다(보수적 추정).

**예상치 못한 보너스:** 코퍼스 확장 후 회귀 여부를 확인하려고 Phase 1의 24개 에이전트
평가셋을 다시 돌렸는데, 통과율이 91.7%→**95.8%**로 올랐다. 그동안 반복적으로 실패하던
"전자레인지+고단백" 조합 케이스가 이번에 새로 추가한 조리환경×취향 조합(combo) 문서
덕분에 통과로 바뀌었다. 코퍼스 확장이 retrieval 지표뿐 아니라 에이전트 최종 품질에도
직접 영향을 준다는 걸 다시 한번 수치로 확인했다 — Phase 1·2·3이 서로 독립된 작업이 아니라
누적되는 구조라는 것이 계속 증명되고 있다.

**배운 점:** "측정 방법을 어떻게 설계할지"가 "측정 결과"보다 먼저 고민해야 할 문제였다.
잘못된 비교 방법(억지 id 매핑)을 쓰면 숫자는 나오지만 의미가 없고, 오히려 그 함정을
알아채고 더 솔직한 질문으로 바꾸는 과정 자체가 이번 Phase에서 가장 가치 있는 결정이었다.

## 2026-06-26 — Phase 4-1: 백엔드 영속화, "모바일 화면을 어떻게 만들지" 전에 웹으로 먼저 검증

**문제:** 로드맵의 Phase 4("백엔드 영속화 + 모바일 화면 실제 구현·연동")는 그동안의 Phase보다
범위가 넓다. DB 스키마, 사용자 구분 여부, API 계약, 화면 흐름까지 한꺼번에 정해야 했고,
사용자도 "모바일 화면을 바로 만들기 전에 웹으로 먼저 확인하고 싶다"고 명시적으로 요청했다.

**결정 1 — 웹으로 먼저 검증하는 전략:** Expo 시뮬레이터는 코드 한 줄 고칠 때마다 리로드가
느리고 디버깅도 불편하다. 반면 백엔드 API 계약(요청/응답 모양)과 화면 흐름은 브라우저에서
훨씬 빠르게 검증할 수 있고, API가 한 번 검증되면 모바일은 거의 같은 fetch 호출을 RN
컴포넌트로 옮기는 작업이 된다. 그래서 "DB 설계 → 웹에서 검증 → 모바일 포팅" 순서로
진행하기로 했다(모바일 포팅은 다음 작업으로 분리).

**결정 2 — 단일 사용자(인증 없음):** 로그인/회원 시스템이 아직 없는데 냉장고 재료를 사용자별로
구분할 방법이 없었다. 두 선택지(단일 사용자 vs. 임의 device id 발급)를 사용자에게 직접 제시해
"일단 단일 사용자로"를 선택받았다 — 포트폴리오 데모 단계에서는 인증 자체가 별도 Phase가 될
만큼 큰 작업이라, 지금 단계에서 무리하게 끼워 넣지 않는 게 맞다고 판단했다(인증은 Phase 5
신뢰성 강화 또는 별도 Phase에서 다룰 후보로 남겨둠).

**결정 3 — SQLModel 선택:** 이 프로젝트는 이미 모든 데이터 모델(`UserProfile`,
`FridgeIngredient` 등)을 Pydantic으로 정의해뒀다. 순수 `sqlite3`(표준 라이브러리, 의존성
추가 없음)도 고민했지만, SQLModel은 Pydantic `BaseModel`을 그대로 테이블로 쓸 수 있어 기존
스키마 정의 방식을 DB까지 자연스럽게 연장할 수 있고, FastAPI와도 표준적으로 같이 쓰이는
조합이라 새 의존성 1개를 추가하는 비용보다 얻는 일관성이 크다고 판단했다.

**결정 4 — `/recommend` API 계약을 바꿈 (하위 호환 깨는 결정):** 기존 `/recommend`는
요청 body에 프로필+재료 전체를 매번 실어 보내야 했다. 하지만 실제 제품 구조(모바일의
냉장고 탭과 레시피 탭이 분리된 화면)를 생각하면, 재료는 냉장고 탭에서 영속적으로 관리되고
레시피 탭은 "지금 냉장고 기준으로 추천해줘"라는 요청만 보내면 된다. 그래서 `/recommend`를
`{today}`만 받고 프로필·재료는 서버 DB에서 읽도록 바꿨다. 이 변경이 기존 클라이언트와 호환되지
않는다는 걸 알고 있었지만, 영향 범위를 확인해보니 `eval/run_eval.py`와 Phase 1~3 평가
하니스는 `naengchae_chain.graph`를 직접 호출하므로(web API를 거치지 않음) 전혀 영향이
없었고, 웹 데모(`index.html`)는 이번 작업에서 같이 고치는 대상이라 문제가 없었다.

**검증:** 브라우저가 없는 환경이라 `curl`도 없어서, Python `urllib`로 서버를 실제로 띄워
전체 흐름(프로필 저장 → 재료 추가 2건 → 냉장고 목록 조회 → `/recommend` 호출 → 삭제 →
존재하지 않는 id 삭제 시 404)을 끝까지 호출해 확인했다. 모든 단계가 기대한 그대로
동작했고, `/recommend`는 DB에 저장된 두부·양파만으로 두부조림/두부김치/두부된장국을
한 번에(재시도 없이) 추천했다.

**배운 점:** "영속화"라는 한 단어 뒤에 사용자 구분, DB 선택, API 계약 변경까지 숨어 있었다.
이런 굵은 단위의 Phase는 시작 전에 작은 결정 여러 개로 쪼개서 각각 이유를 명시하지 않으면,
나중에 "왜 이렇게 했지"를 설명할 수 없는 코드가 나온다.

## Phase 4-2: 웹 데모 실사용 후 발견한 4가지 UX/데이터 문제 (2026-06-26)

Phase 4-1에서 만든 웹 데모를 실제로 써보면서 사용자가 4가지 문제를 발견했다. 모바일 화면을
만들기 전에 웹에서 검증하기로 한 전략(Phase 4-1 결정 1)이 의도대로 작동한 사례 — 같은
문제를 모바일(Expo)에서 다시 발견했다면 디버깅 주기가 훨씬 길었을 것이다.

**문제 1 — 유통기한 입력 필드가 무엇을 뜻하는지 불명확:** "유통기한"이라는 라벨만 있고
설명이 없어서, 구매한 날짜인지 먹어야 하는 날짜인지 헷갈렸다.
**해결:** 날짜 입력 위에 "이 재료를 언제까지 먹어야 하는지의 날짜입니다 (구매한 날짜가
아닙니다)"라는 명시적 설명 문구(`field-hint`)를 추가했다. 코드 로직(`daysUntil`,
`expiry_date` 컬럼 등)은 원래부터 "먹어야 하는 날짜"로 일관되게 동작하고 있었으므로,
이건 로직 버그가 아니라 순수 UI 커뮤니케이션 문제였다.

**문제 2 — "추가" 버튼이 마우스 위치에 따라 눌리다 안 눌리다 함:** 원인을 찾아보니
`재료명 입력 / 날짜 입력(고정 140px) / 추가 버튼`이 한 줄에 flex로 들어가 있었는데, 카드
너비(380px - 패딩)에서 셋을 한 줄에 배치하면 여유 공간이 빠듯해서, 브라우저(Chrome)의
네이티브 `<input type="date">` 캘린더 아이콘 렌더링 영역이 버튼 쪽으로 살짝 겹치는
구간이 생겼다. 버튼의 어느 지점을 눌렀는지에 따라 클릭이 date input으로 먹히기도 하고
버튼으로 먹히기도 한 것 — 사용자가 관찰한 "마우스 위치 의존성"과 정확히 일치하는 설명이다.
**해결:** 재료명 입력을 한 줄(전체 너비)로 분리하고, 날짜 입력+추가 버튼을 별도의 둘째 줄로
내려서 셋을 한 줄에 욱여넣지 않도록 구조를 바꿨다. 추가로 `.add-btn`에 `flex-shrink:0`을
줘서 버튼이 다시 줄어들 가능성 자체를 차단했다. 근본 원인이 "네이티브 date input의 정확한
렌더링 폭"이라 디버깅하기 까다로웠는데, 레이아웃을 분리하면 원인 규명 없이도 증상이 사라져서
이쪽을 택했다.

**문제 3 — 가구 형태를 "1인 가구"/"핵가족" 두 칩 중에서 고르게 했더니 2~3인 가구가 애매:**
원래 UI는 가구원 수 입력칸을 "핵가족"을 선택했을 때만 보여줬는데, 이러면 사용자가 먼저
모호한 카테고리(핵가족이 몇 명부터인지 불명확)를 골라야 가구원 수 입력칸이 나타나는
순서가 거꾸로였다.
**해결:** 가구 형태 칩 자체를 없애고, 가구원 수 입력(1~5명, 5명은 "5명 이상")만 항상
보여주도록 바꿨다. `householdType`(single/family) 값은 서버 모델/프롬프트가 이미 이
필드를 쓰고 있어서 스키마는 그대로 두고, `memberCount === 1 ? 'single' : 'family'`로
프론트엔드에서 자동 계산해 보내도록 했다 — 사용자에게 보이지 않는 내부 파생값으로 전환.
백엔드 모델(`UserProfile.householdType`)을 건드리지 않고 프론트엔드 한 곳만 고치는 게
범위가 가장 작은 수정이라고 판단했다.

**문제 4 — 레시피에 재료 "수량"이 전혀 없어서 가구원 수에 비례한 분량 조정이 불가능:**
이게 네 문제 중 가장 범위가 컸다. 기존 `Recipe.usedIngredients`/`missingIngredients`는
재료 이름만 있는 `list[str]`이라, 인원수에 맞춰 "양을 조정"한다는 게 애초에 표현할 수 없는
데이터 구조였다(LLM이 "2인분"이라고 `servings`만 적어도 그게 맞는지 검증할 방법이 없었다).
**해결 — 단계별:**
  1. `models.py`에 `UsedIngredient(name, amount, perServingAmount)` /
     `MissingIngredient(name, amount)`를 새로 정의하고 `Recipe`의 두 필드 타입을
     `list[str]` → 이 모델들의 리스트로 바꿨다. `missingIngredients`에는
     `perServingAmount`를 넣지 않았다 — 모자란 재료는 "장보기 목록"의 의미라 1인분 환산이
     필요 없다고 판단.
  2. `prompts.py` 규칙 5에 "servings는 항상 memberCount와 동일해야 한다", "amount는
     servings 인분 전체 기준, perServingAmount는 amount ÷ servings"를 명시했다. 이
     프로젝트의 기존 패턴(LLM 출력 형식을 프롬프트에 정확히 명시 + 코드로 재검증)을
     그대로 따른 것.
  3. `graph.py`의 `_validate`에 `recipe.servings != member_count`면 issue를 추가하는
     검증을 새로 추가했다 — "LLM이 그렇다고 말했다"를 믿지 않고 결정론적으로 다시
     확인하는 이 프로젝트의 핵심 패턴(Phase 1부터 일관)을 새 필드에도 그대로 적용했다.
     `usedIngredients`가 객체가 됐으므로 `_ingredient_match`/`_ingredient_match_any`도
     `.name`을 보도록 수정했다.
  4. 1차 테스트에서 `memberCount=1`로 호출했을 때 "김치 1/4모"처럼 재료에 안 맞는 단위
     (모=두부 단위)를 쓰는 경우를 발견했다 — 수량 자체(분수/소수 변환)는 정확했지만 단위
     선택이 틀린, 새로 생긴 필드 특유의 품질 문제였다. 프롬프트에 "재료 종류에 맞는 단위를
     쓰라"는 규칙과 "다른 재료의 단위를 가져다 쓰지 말라"는 반례를 추가해 재검증했고,
     이후 호출에서는 두부=모, 계란=개, 김치=g로 정상화됐다.
  5. `web/main.py`에 `UsedIngredientOut`/`MissingIngredientOut` 응답 모델을 추가하고,
     `index.html`의 레시피 카드 렌더링에 "재료명 + 수량 + · 1인분 환산수량"을 표시하도록
     수정했다.

**검증:** `memberCount=4`로 추천을 호출해 `servings=4`, 계란 `amount=4개`/
`perServingAmount=1개`처럼 정확히 4배수 관계가 나오는지 확인했고, `memberCount=1`로도
같은 흐름이 1배수로 일관되게 나오는지 재확인했다(amount == perServingAmount일 때 정상).
두 경우 모두 `valid=true`, `retryCount=0`으로 첫 시도에 통과했다.

**왜 eval 하니스(Phase 1~3)가 깨지지 않았는지:** `eval/run_eval.py`는 `final_state`의
`valid`/`feedback`/`retry_count`와 `recipe.name`만 읽고 `usedIngredients`의 내용은
채점에 쓰지 않는다. 모델 변경 전에 이 사실을 먼저 확인한 뒤 진행했다 — 영향 범위를 안
재고 구조를 바꾸면 어디서 깨질지 알 수 없기 때문에, 이번에도 "고치기 전에 먼저 누가 이
데이터를 쓰는지 확인" 순서를 지켰다.

## Phase 4-3: 유통기한 입력을 "정확한 날짜" 대신 "임박 여부 토글"로 단순화 (2026-06-26)

Phase 4-2에서 날짜 입력란에 설명을 추가했지만, 사용자가 다시 써보고 "재료마다 정확한
소비기한을 다 입력하는 게 번거롭다, 그냥 곧 먹어야 하는지 여부만 체크하게 하면 어떻겠냐"는
제안을 줬다.

**검토한 선택지:**
- (A) 순수 바이너리 — 날짜 개념을 완전히 없애고 "임박/안임박" 불값만 저장.
- (B) 절충안 — 기본 입력은 토글(켜면 "곧 먹어야 해요")이지만, 토글을 켰을 때 내부적으로는
  여전히 날짜(오늘 날짜)를 채워서 저장하고, 정확한 날짜를 아는 경우를 위한 선택적 입력칸을
  접어서 남겨둠.

(A)는 입력이 가장 단순하지만 두 가지를 잃는다: 여러 재료가 동시에 "임박"일 때 상대적
긴급도를 구분할 수 없고(날짜 기반이면 D-1이 D-3보다 급하다는 걸 알 수 있는데 불값은
구분 불가), 날짜는 시간이 지나면 자동으로 더 긴급해지는데 불값은 사람이 갱신하지 않으면
영원히 그 상태로 고정된다. (B)를 택하면 이 두 가지를 잃지 않으면서 입력 마찰만 없앨 수
있다 — 토글 on을 "오늘 날짜로 저장"으로 변환하면 `chain.py`의 `daysUntil`/`D+n`
배지·`EXPIRY_WARNING_DAYS` 검증 로직을 단 한 줄도 안 건드리고 그대로 재사용할 수 있다는
것도 (B)가 작은 변경으로 끝나는 이유였다. 사용자에게 두 옵션의 트레이드오프를 설명하고
(B)를 선택받았다.

**구현:** `index.html`의 날짜 입력을 기본 화면에서 빼고, 그 자리에 칩 스타일 토글 버튼
("🔥 곧 먹어야 해요")을 넣었다. 정확한 날짜를 아는 사용자를 위해 "정확한 소비기한을
아시나요? (선택)" 링크를 두고, 누르면 날짜 입력칸이 펼쳐지며 거기에 값이 있으면 토글보다
우선 적용된다(`addIngredient()`에서 `expiryEl.value || (urgentOn ? today() : null)` 순서로
판단). 백엔드(`db.py`, `models.py`, `chain.py`)는 전혀 건드리지 않았다 — `expiryDate`
필드의 의미(스키마)는 그대로고, 그 값을 어떻게 만들어내는지만 프론트엔드에서 바뀐 것이라
변경 범위가 한 파일(`index.html`)에 한정됐다.

**검증:** 토글 on 상태로 추가했을 때 보내지는 값과 동일한 `expiryDate=오늘 날짜`를 API에
직접 호출해 DB에 정상 저장되고 `/fridge` 조회에 그대로 반영되는 것을 확인했다(브라우저 JS
자체는 헤드리스 브라우저가 없는 환경이라 직접 실행해볼 수 없어, 토글이 만들어낼 값과 동일한
입력으로 백엔드 경로를 검증하는 방식을 택함).

## Phase 4-4: 모바일 FridgeScreen/RecipeScreen을 실제 백엔드에 연동 (2026-06-26)

웹 데모로 API 계약과 데이터 흐름을 충분히 검증했으니(Phase 4-1~4-3), 같은 백엔드를 모바일
화면에 연동했다. 원래 `FridgeScreen.js`/`RecipeScreen.js`의 코드 주석에는 "하드코딩 레시피
10개로 채울 예정"이라고 적혀 있었는데, 그 사이 LangGraph 에이전트 + DB 백엔드가 이미
완성됐으므로 계획을 하드코딩 대신 실제 API 연동으로 키웠다 — 이렇게 계획이 바뀐 이유를
기록해두지 않으면 나중에 "왜 하드코딩이 아니라 API 연동이 됐지"를 설명할 수 없기 때문에
남긴다.

**결정 1 — API 주소를 코드에 박지 않고 `EXPO_PUBLIC_API_URL` 환경변수로 분리:** 모바일
앱은 폰/에뮬레이터에서 실행되므로, 백엔드(FastAPI)에 접속하는 주소가 실행 환경마다 다르다
(iOS 시뮬레이터는 `localhost`, Android 에뮬레이터는 `10.0.2.2`, 실기기는 PC의 LAN IP).
사용자가 아직 어떤 환경에서 테스트할지 정하지 않은 상태였으므로, 이 결정을 나중으로 미룰 수
있게 `src/utils/api.js`에서 `process.env.EXPO_PUBLIC_API_URL` 하나만 읽어서 쓰도록
분리했다. Expo SDK 56 공식 가이드(환경변수 문서)에 따라 `EXPO_PUBLIC_` 접두사 변수는
빌드 시 그대로 인라인되는 방식이라 추가 설정 없이 동작한다. 실제 값은 `.env.example`
(커밋됨, 안내문 + localhost 기본값)을 `.env.local`(이미 `.gitignore`의 `.env*.local` 규칙으로
무시됨)로 복사해서 넣도록 안내해뒀다 — 값을 정하기 전까지는 기본값(localhost:8000)으로
그냥 동작한다.

**결정 2 — 온보딩의 가구 형태 칩을 모바일에서도 동일하게 제거:** 이번 작업의 범위는
원래 FridgeScreen/RecipeScreen이었지만, 온보딩(`StepHousehold.js`)에 웹에서 이미 고친 것과
완전히 같은 버그(1인가구/핵가족 두 칩 중 고르게 해서 2~3인 가구가 애매한 문제)가 그대로
남아 있었다. 같은 버그를 모바일에만 남겨두면 일관성이 깨지고, 사용자가 결국 모바일에서도
같은 불편을 겪을 게 명확했으므로 범위를 넓혀 같이 고쳤다. 가구원 수 슬라이더(기존에 이미
"가족" 선택 시에만 보이던 컴포넌트)를 항상 보이게 하고, `householdType`은
`memberCount === 1 ? 'single' : 'family'`로 `OnboardingScreen.js`에서 자동 계산한다 — 웹의
해법과 동일한 패턴.

**결정 3 — 온보딩 완료 시 AsyncStorage(로컬)뿐 아니라 백엔드 `/profile`에도 동기화하되,
실패해도 온보딩을 막지 않음:** 모바일 앱의 "온보딩 완료 여부" 판단(`RootNavigator.js`)은
AsyncStorage만 보고 결정되는 기존 구조를 그대로 유지했다(오프라인에서도 앱을 열 수 있어야
하므로). 하지만 `/recommend` API는 서버 DB의 프로필을 읽으므로, 온보딩 완료 시
`saveProfile()`로 서버에도 보내야 한다. 다만 이 시점에 네트워크가 없거나
`EXPO_PUBLIC_API_URL`이 아직 맞지 않게 설정돼 있을 수 있으므로(사용자가 "나중에 정하겠다"고
한 바로 그 이유), 이 동기화를 실패해도 온보딩 자체를 막지 않고 `try/catch`로 감쌌다. 대신
`api.js`에 `ensureProfileSynced()`를 만들어, 레시피 추천을 누르는 시점에 한 번 더
"서버에 프로필이 있는지" 확인하고 없으면 로컬 값으로 재시도하도록 안전망을 뒀다 — 한 번의
실패가 영구적인 깨짐으로 이어지지 않게 하는 패턴.

**결정 4 — 유통기한 토글(Phase 4-3)을 모바일에도 동일하게 적용하되, "정확한 날짜" 보조
입력은 모바일에서는 생략:** 웹에서는 정확한 날짜를 아는 사용자를 위해 접을 수 있는 날짜
입력칸을 남겨뒀지만, React Native에는 표준 날짜 선택 UI가 없고(`@react-native-community/
datetimepicker` 같은 네이티브 의존성을 새로 추가해야 함) 이 작업의 핵심 범위가 아니라고
판단해 모바일은 "곧 먹어야 해요" 토글만 제공한다. 토글 on이면 `today()`를 `expiryDate`로
보내는 동작은 웹과 동일해서 백엔드 호환성에는 문제가 없다. 필요해지면 나중에 네이티브
날짜 선택기를 추가하는 작업으로 분리하면 된다.

**검증 방법과 한계:** 이 샌드박스에는 휴대폰·에뮬레이터·시뮬레이터가 없어 실제 화면을
탭해볼 수 없다. 대신 (1) `conda`로 Node.js를 설치하고 `npm install`을 실제로 실행했고,
(2) 프로젝트의 `@babel/core` + JSX 플러그인으로 새/수정 파일 5개를 전부 직접 파싱해
문법 오류가 없는지 확인했고(일부러 깨뜨린 코드로 이 검증 방법이 실제로 오류를 잡아내는지도
먼저 확인함), (3) `npx expo export --platform android`로 앱 전체(928 모듈)를 실제로 Metro로
번들링해 Hermes 바이트코드까지 생성하는 데 성공했다 — import 경로, JSX, 컴포넌트 export
등 정적으로 잡을 수 있는 오류는 이 단계에서 다 걸러진다. 다만 이건 "빌드가 된다"는 것의
증명이고, "버튼을 눌렀을 때 실제로 잘 동작한다"는 것의 증명은 아니다 — 실제 기기/에뮬레이터
테스트는 사용자가 `EXPO_PUBLIC_API_URL`을 본인 환경에 맞게 설정한 뒤 직접 해야 한다.

## Phase 4-5: 실기기(iPhone) 연동 시도 — 네트워크 터널 + Expo SDK 다운그레이드 (2026-06-26)

사용자가 실제로 본인 iPhone의 Expo Go로 접속을 시도하면서 두 가지 문제를 차례로 만났다.

**문제 1 — 이 환경은 사용자 휴대폰과 같은 네트워크가 아님:** 이 Claude Code 세션은
원격/클라우드 서버(`instance-18854`, 내부 IP `172.17.91.182`)에서 돌고 있어서, 휴대폰과
같은 와이파이를 쓰는 "LAN IP" 방식(애초에 `.env.example`에 안내해둔 방법 중 하나)이 원천적으로
불가능했다. 사용자에게 실행 환경을 먼저 확인받은 뒤(로컬 PC가 아니라 원격 서버임을 사용자가
직접 확인), 두 개의 별도 터널이 필요하다고 판단했다 — ① 모바일 앱이 부르는 FastAPI 백엔드
(포트 8000)용 터널, ② Expo Go가 JS 번들을 받아오는 Metro 개발 서버(포트 8081)용 터널. 이
둘은 서로 다른 포트/서비스라 하나의 터널로 묶을 수 없다.
- 백엔드용: `cloudflared`(Cloudflare Quick Tunnel)를 GitHub 공식 릴리스에서 직접 받아 썀.
  ngrok 대신 이걸 고른 이유는 ngrok이 최근 정책상 무료 사용도 계정+authtoken 가입이 필요해진
  반면, Cloudflare의 "Quick Tunnel"은 가입 없이 즉시 임시 URL을 발급해주기 때문 — 이번
  목적(짧은 1회성 테스트)에 더 적합.
- Metro용: `npx expo start --tunnel` (내부적으로 `@expo/ngrok` 사용, devDependency로 추가).
- 두 터널 다 만든 뒤, 내가 먼저 `urllib`로 외부에서 실제로 호출해보고(백엔드는 `/profile`
  응답 200, Metro는 매니페스트 JSON 200) 사용자에게 넘기기 전에 검증했다 — "터널이 생성됐다"는
  로그만 믿지 않고 실제로 도달 가능한지 직접 확인하는, 이 프로젝트에서 계속 반복된 "주장을
  코드로 재검증" 패턴을 인프라 설정에도 적용한 것.
- Expo Go 앱에 "Enter URL manually" 버튼이 있을 거라고 처음에 안내했는데, 실제로는 그 기능이
  일반 Expo Go가 아니라 커스텀 개발 빌드(dev-client)에만 있다는 걸 사용자 피드백으로 알게 됐다
  (웹 검색으로 재확인). 대신 `exp://` 스킴은 휴대폰 브라우저 주소창에서 열어도 Expo Go로
  넘어가는 보편적인 동작이라 그 방법으로 안내했다.

**문제 2 — "Project is incompatible with this version of Expo Go":** 사용자가 방금 새로
설치한 최신 Expo Go에서도 이 에러가 났다. 원인을 찾아보니 내 설정 문제가 아니라 Expo/Apple
쪽의 실제 이슈였다: 2026년 5월부터 Apple App Store의 Expo Go 심사가 밀려 있어서, 스토어에
배포된 Expo Go가 SDK 54에 멈춰 있는 상태였다(SDK 55도 심사 대기 중, 타임라인 없음). 이
프로젝트는 SDK 56을 쓰고 있어 스토어 버전 Expo Go와 호환되지 않았다.
- 옵션을 검토했다: (a) 프로젝트 SDK를 54로 다운그레이드, (b) `eas go`/개발 빌드(Apple
  Developer 유료 계정 필요), (c) TestFlight 외부 베타. 사용자가 유료 Apple Developer
  계정이 없는 iOS 사용자라고 확인한 뒤, 무료로 당장 테스트 가능한 (a)를 진행했다 — 이건
  Expo 공식 changelog가 바로 이 상황(앱스토어 Expo Go 지연)에 대해 권장하는 방법이기도 하다.
  의존성을 실제로 낮추는 변경(react 19.2→19.1, react-native 0.85→0.81)이라 진행 전에
  사용자에게 한 번 더 확인받았다.
- 절차: `npm install expo@^54.0.0` → `npx expo install --fix` → `npx expo-doctor`로 검증.
  중간에 `expo-doctor`가 `expo-font` 중복 설치(구버전 잔여 + SDK54 버전 공존)를 잡아내서
  구버전을 지우고 `npx expo install expo-font`로 다시 정리했다 — 18/18 체크 통과 확인.
  `npx expo export --platform ios`로 다시 한 번 실제 빌드 성공을 확인한 뒤에야 사용자에게
  새 링크를 전달했다.

**`exp://` 링크 전달 방법 — 시도와 실패:** `exp://` 같은 커스텀 스킴은 모바일 브라우저
주소창에 직접 타이핑하면 검색어로 처리되는 경우가 많았다(주소창 자동완성 휴리스틱 문제).
"메모 앱에 적으면 iOS가 자동으로 링크로 인식해줄 것"이라고 안내했는데, 실제로는 iOS의
텍스트 자동 링크 인식이 `http(s)://`/`mailto:` 등 표준 스킴 위주이고 `exp://` 같은
커스텀 앱 스킴은 자동 인식 대상이 아니었다 — 잘못된 가정이었다. 사용자가 결국 다른
방법으로 스스로 연결에 성공했다(정확한 방법은 대화에 남지 않음). 교훈: 커스텀 URL
스킴을 사람에게 전달할 때는 "타이핑"이나 "메모 앱 링크화"에 의존하지 말고, 애초에
QR코드 이미지로 주는 게 가장 신뢰도가 높다 — 다음에 비슷한 상황이 오면 처음부터 QR
이미지 생성을 시도할 것.

**검증 완료:** 사용자가 실제 iPhone에서 Expo Go로 접속해 온보딩 → 냉장고 탭 → 추천 탭까지
직접 눌러보고 정상 동작을 확인했다. Phase 4(백엔드 영속화 + 모바일 연동)의 핵심 목표였던
"웹에서 검증한 것과 같은 백엔드로 실제 모바일 화면이 동작한다"가 실기기 기준으로 완료됐다.
임시 터널(계정 없는 Cloudflare Quick Tunnel + Expo `--tunnel`)은 안정성 보장이 없는
일회성 테스트 편의이고, Phase 6(배포)에서는 실제 호스팅으로 교체될 것이다.

## 이 프로젝트의 핵심 차별점 (다시 명확히)

이 프로젝트는 "RAG 에이전트를 만들어봤다"에 그치지 않는다. 목표는:

1. **MLOps/LLMOps 파이프라인 구축** — 평가(eval) → 관찰성(observability) → 신뢰성(테스트/폴백)
   → 배포까지, LLM 애플리케이션을 운영하는 데 필요한 인프라적 요소를 단계별로 직접 설계하고
   연동하는 것
2. **실제 동작하는 애플리케이션** — 모바일 앱 + 백엔드 + 배포가 끝까지 연결된 완성품으로
   만드는 것
3. **문제 해결 과정 자체를 기록** — 이 문서처럼, 무엇이 문제였고 어떤 전략으로 풀었는지를
   결과물만큼 중요하게 남기는 것

"통과율을 측정해서 개선했다"는 위 1번(LLMOps)의 한 조각(평가 단계)일 뿐, 프로젝트 전체의
차별점은 아니다.
